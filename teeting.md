# Teeting
Text + Meeting = Teeting。

## 定義と、ここでやりたいこと
from [コミュニケーションにおける「3つの拘束」](3_communication_restraint.md)

定義はまだ甘いが、いったん引用しておく。

- > 筆者が提案するやり方として Teeting がある。これは Text + Meeting から取ってきた造語であり、「制限時間の間」「参加者全員が」「n個の議題が書かれているエリアを自由に編集」することで議論を行う会議形態である。つまり **どこから参加してもいいし、どの話題に書き込んでも構わないが、会議時間だけは守れ（そして時間内に全議題について結論を出すよう立ち回れ）** というものだ。
- > Teeting とは Text + Meeting からつくった造語であり、言わば筆談会議である。フロック型ツール上で、制限時間以内に、参加者全員が各自自由に編集を行い、しかし結論を出すというものだ。無論、何のお膳立ても無しに行えるはずはないから、何らかの方法論を整え、仕組みとして実装する必要があろう。

やりたいことはこれ。

> 無論、何のお膳立ても無しに行えるはずはないから、何らかの方法論を整え、仕組みとして実装する必要があろう。

ここで整えていく。

## 共通動線をどう確保するか
teetingは「各自が自由に好きな話題にリーチしている」が、自由ゆえに収拾をつけづらい。かといって露骨に制御に走ればteetingの意味がなくなる（従来どおりの全拘束会議になる）。

このジレンマを突破するためのアプローチとして **全員が高確率で通る動線（共通動線）** をつくり、ここに指示命令お願いなどを書いておくことで「まあそのうち読んでもらえるでしょう」を狙う、というものがある。

共通動線案1: ロビーモデル

- ホテルのロビー、あるいはデザインパターンでいう Mediator のようなものをつくる

```
ネットワーク型のデータ構造だと、たとえばこうか。
必ずロビーに行けるようになっている。

    o---+
        |
        o  o----o----o
        |  |
        |  |
   o----LOBBY---o
   |     | |
   |     | +--------o---o
   |     o----o
   |          |
   o----------o

階層的なデータ構造だと、こうだろうか
ロビーから根を下ろしている（ので、上へと辿っていけば必ずロビーに着ける）。

  LOBBY

```

## Topic と Purpose を使い分ける
トピック指向。ある話題Aは、Aについて書いたページPAでのみ行なえ。PAにはA以外のことを書かない or 書いてもいいがすぐ別のページに分ける。Aに関する経過も、全部PAにぶら下げる。 / 単位は「ページ」に限らない。たとえばページ内の「箇条書き（にぶら下げられた塊）」かもしれない。

パーパス指向。ある目的Aは、A用の手段であるWAを使って満たせ。Aを満たすのにWA以外を使うな。ここでいう目的とは「この会議の目的」「このコミュニケーションの目的」に等しい。つまり「会議」「コミュニケーション」などと雑にくくって適当なやり方でやる、のではなく「こういう目的があって、そのためにはこういう手段を使えばいい」という風に最適化する。

手段とは制約。フォーマットやプロセスやプロトコル。こういうやり方で、こういう流れで、こういう書式でやり取りしましょうみたいなもの。

トピックは必要に応じてつくる。議題がnつあったらn個つくるべき。またパーパス指向レベルのものもある。たとえば「2021/10/17 13-15の会議において、参加者全員に言いたいことを伝えるページ」もトピックの一つだし、「2021/10/17 13-15の会議において、参加者全員に可及的速やかに共有したいことを伝えるページ」もトピックである。もちろん後者の場合、可及的速やかに伝えるための何らかの仕組みは必要（口頭だとその場で声上げれば済むが、teetingは参加者がどのトピックにいるかわからん）。

★パーパス。オレオレ語でいうと「コミュニケーションシップ」が一番正確なニュアンス。が、こんな言葉使っても皆わからん。パーパスの方がまだわかりやすいか？

## PIDS
Putから始まるやり方。

- Put / 並べる …… 元ネタを置く。議題、持論、質問 etc
- Insert / 混ぜる …… 誰かの意見に自分の意見を差し込む。コメント。返信。やり取り。
- Decide / 決める …… 決定事項を決める。意思決定が必要。「そんなの聞いてねえぞ」にならないためには拘束が必要（事後でいいならその了承）。ただし「後で必要に応じて変えられる（変えやすい）」なら別にいい（変えるためのやり取りをまたすればいい）。
- Summarize / まとめる …… 並べたこと、混ぜたこと、そして特に決めたことをあとで参照しやすくするためにまとめる。決定事項によっては文書化を行うことも。

Decide と Summarize の違い

- Decide は、Insert までされた「まとまってない状態」でも行える
- Summarize は、（決定事項も含めて）まとまってない状態をまとめに行く
- Q: PISD ではダメなのか
    - Ans: ダメなことはないが、まとめる力は資質であり万人にできることではないので推奨しない
    - しかし、決めるためにまとめる必要がある場合もあるので、まあよしなに
    - ここでは「PIDSでもPISDでもいい」「というかP I D Sは全部行ったり来たりする」としておく
- Q: 行ったり来たりの対象として P が含まれているが、どういうことか？
    - たとえば「新しく出てきたこの論点は、いったん手元で深堀りしたいな」ってとき
        - この場合、自分エリアで深堀りすることになる（Put）

どこから teeting を行うか

- putから行う
    - 対面会議だとブレストみたいに「まずは一人で出して抱える」「それをターン制で出す」
    - が、teetingだと「いきなり皆に見える場所に出せる」「別に最初は手元でやってもいいけど、putフェーズの間に出しておかないと以後扱われないよ」
- insertから行う
    - 読み込みの時間が必要そう？
- decideから行う
    - これ、読み込みかなり必要そうだが……。要するに put と insert でカオスった戦場跡地を見て決定しなきゃいけない
        - 意思決定者にそんな余裕はない
        - 気概もないだろう（意思決定者はコンテキストと非言語情報を大事にするので、自分が参加してそれらを持てなかったシチュでは慎重になる。せいぜい参考にしかしない）
    - とすると❌（decideから行う、はありえない）
- summarizeから行う
    - まとめ作業のペアワーク、モブワークともいえる
    - n人で「これはどうまとめようか」「ここにこうまとめようか」「いいんじゃない？」「いやこっちがよくない？」
    - ワーキングトピックをつくると良い
        - まとめたい対象をn個を並べたページ
        - 1つずつ処理していく
        - あるいは「Aさんはこれとこれ、Bさんはこれとこれ、で、30分後にいったん戻ってきて休憩はさんだ後、レビューしよう」みたいなやり方でも良いけど

★読み込み(Read)を入れると PRIDS。「ぷらいどす」となって読みやすさ良い感じ。

## サマリー駆動
- 「ここまで議論した内容をまとめるとこうです」と要約をはさんでいったん仕切り直す、という進め方
- Q: これをしないと？
    - やり取りが延々と続く
        - 何が本質なのかが「各自の頭の中」にある状態で留まっている
        - 当然ながらやり取りの量・密度・時間が増えるほど、各自の乖離は広まっていく
    - 認識齟齬が肥大化していく
    - 結果、噛み合わなくなってくる
    - だからサマリーをつくって区切る（肥大化を止める）

思想

- 「途中経過をサマライズすることさえできないのに、議論を続けるなんてできると思うか？」
- サマライズできない≒サマライズできるほど理解できてない≒十分に議論できてない
    - つまりサマライズできてない≒そもそも十分に議論できてない
- ぐだぐだは許さんぞ
    - サマリーという「目に見えるもの」を必ず出せ

## コミュニケーションの直接性 ～DIS～
- Direct
    - 直接やり取りする
- Indirect
    - 間接的にやり取りする
    - 共同エリアにメッセージを置く、と言っても良い
- Semi-direct
    - 時間差でやり取りする
    - 相手の動線上にメッセージを置く、と言っても良い

コミュニケーションというと D をイメージしがちだが、Teeting では D をしない。

Scrapbox は I のみが存在する世界。だがリアルタイム性を担保できないので結局チャットも必要になる。チャットをフロー型なので廃したい。どうすれば？動線に置けばいいじゃないか、は一つのアイデア。これを S として整理した。

上述の共通動線は、I に相当する。共同エリアが皆の動線になった、と捉える。

## 全体言及と部分言及
- 全体/部分
    - 全体に対して言及する
        - 書名を取り上げてあーだこーだ言うのはこれ
    - 部分に対して言及する
        - 一文を引用してここについてはこーだあーだと言うのはこれ
    - 両者は混ざり合う
        - 書名を取り上げてあーだこーだする中で、一文を引用してあーだこーだもする
        - 一文を取り上げてあーだこーだする中で、そもそもこの本は……と全体を取り上げた論調もする

全体言及は良くも悪くも雑。緩い。

部分言及は良くも悪くも細かい。きつい。

いずれにせよコンテキストロス（コンテキストを読まずor読めずに正しく読めない）は起こり得る。全体言及だと細部見てないとか。部分言及だと前後見てないとか。

## Meeting Driven Working
会議駆動仕事。

とりあえず会議をぶちこんで、その中で仕事しよう（決めよう）というもの。忙しい意思決定者や中間管理職は大体これになる。

パターン

- パターン1: 有能で処理能力が高いが、何をどの順番ですればいいかまではわからない
    - だから部下達に「私の予定表にぶち込め」とする
        - というか、そうするしかない
    - 打ち込まれたものを片っ端からさばいていく
- パターン2: 無能な上層部と、上層部の部下となる中間管理職
    - 上層部の特徴
        - 対面会議しか手段を知らない
        - 人付き合いの世界を生きている（権限持ってる人を動かすには「嫌われない」「できれば好かれる」）
    - ゆえに彼らは対面会議ばかりする
    - 部下である中間管理職達は巻き込まれる
    - そして権限は管理職以上に集中しているので、決め事（会議）そのものが多い

## Editor
- 同時編集
    - 無いときつい
    - これがないと「誰々が編集してるので待て」か「編集が競合してるので解決してください」になる
- WYSIWIG
    - マジョリティは装飾に毒されている＆マークアップを知らないので、これに頼りがち
    - 文字装飾は要らない
        - 「個人の主観に基づいた重要度の表現（押し付け）」にすぎないので、打ち合わせには不要
    - 見出しも微妙
        - 文書や資料には必要だが、打ち合わせ時に書く議事メモとしては不要
        - 装飾と同様、押し付けにすぎない
    - そもそも内容と形式を分離できてないという欠陥がある
        - あるいはデータ構造として分離できていても、執筆者の意識としては分離できてない
- plain markup
    - マークアップ言語を完全に手で書く系統（プレビュー機能があったりボタンから文法を挿入したりはある）
    - 一人で書く分には、これが生産的になりつつある
    - 同時編集も備えたエディターだと、この系統は見かけない
        - マークアップをそのまま読むのは辛いから
        - 特に他人が書いたもの
    - Q: プレビュー画面はmust？
        - ans: いいえ
        - プレビュー画面を常に見ている人は、形式にとらわれているだけ
        - まずは内容だけを一通り書くのが早いのに、いちいち「今書いた内容がどう表示されるか」も見ている
            - マルチタスク的
        - もっともこのマルチタスクが一概に悪いとは言えない
            - 書いたことを「実際の表示内容」で眺めて、その場で間違いをfixしながら進めていける
            - ボトムアップというか足元から固めていく
- Drop-WYWIWIG textarea
    - 基本的に WYWIWIG だが、自分のカーソルが乗っている行だけはマークアップになる方式
        - Scrapboxもこれ
    - 読みやすい「レンダリング後の内容」がデフォでありつつも、自分が書くところは plain、なのでいいとこ取りできている
- undo(Scrapbox)
    - 復旧手段
        - ctrl + z
        - page history（ある程度でかい単位で差分管理されてる）
        - backup（バックアップしたデータから復旧できる）
        - recent deleted（ゴミ箱からもとに戻す）
    - 同時編集が絡む場合、自分が編集した部分だけ戻す
        - Undo targetが（他人の編集によって）無い場合は無視する
        - 参考: [Scrapboxのundo/redo - guiland](https://scrapbox.io/guiland/Scrapbox%E3%81%AEundo%2Fredo)
- 行指向
    - 行指向で扱うのが潮流
        - Scrapboxはデータ構造からしてそうなっている
        - OneNoteでも更新者が記録されるのは行単位
    - コンピュータで扱いやすいから＆人間にも直感的にわかりやすいから
    - :memo: 「n行の塊」を扱うモデルもありえる
        - あとで深堀りする

## about 汎用、use Phrase Network
teeting の手段としては、筆者は Scrapbox をベースにして考えている。が、これ以外のアプローチもあろう。たとえば Scrapbox は行指向なので、行指向ではないデータ構造について考えてみることでなにかひらめくかもしれない。

で、早速ひらめいたのがこのモデルだった。phrase networkとでも呼べばいいのだろうか。

### 基本概念
- phrase という単位
    - TwitterでいうTweetみたいなもの
    - 「短文」くらいの単位
- phrase と phrase は繋げることができる
    - 連接
    - 例示
    - 根拠
    - メリット・デメリット etc
- この繋げる部分を「関連(relation)」という
- グラフ理論で言えば、
    - phrase がノード
    - relation がエッジ

```
(phrase) ----------> (phrase)
          relation

```

やり取りするためには？

- たとえば以下
    - 他人の phrase に、自分の phrase を繋げる
    - 話題 X に関する phrase をつくり、A について言及したい人が各自の phrase を A に繋げる


```
反論したいなら、

    (B's phrase) ----------> (A's phrase)
                    反論

補足したいなら、

    (B's phrase) ----------> (A's phrase)
                    補足
```

```
話題XについてA、B、C、Dさんが言及している

    (A's phrase)
        |
        |
        |
        V
    (Topic-X) <------(B's phrase)
      A  A
      |  |
      |  +-----------(D's phrase)
      |
    (C's phrase)


話題Xの情報量が多い場合は、その補足phraseを繋げば良い

                  (補足phrase2)
                      |
                      |
                      |
    (A's phrase)      |
        |             V
        | +-------(Xの補足) <------(補足phrase1)
        | | 
        V V
    (Topic-X) <------(B's phrase)
      A  A
      |  |
      |  +-----------(D's phrase)
      |
    (C's phrase)

```

### Q: Scrapboxと何が違う？
- Scrapboxは
    - Pageというノード
    - Page内は複数人が編集できる
    - Page内からリンクを張ることで繋がる
        - Page内コンテンツとして接続を表現する
        - 別の言い方をすると「エッジ」というデータ構造がない
    - エッジ自体に意味はない
        - 厳密に言えば `#hashtag` と `[link]` は別物扱いされているが
- このモデルは
    - Phraseというノード
    - Phraseは **本人しか編集できない**
    - PhraseからPhraseへの接続は、専用の操作で行う
        - 別の言い方をすると **「エッジ」というデータ構造がある**

つまり、

- Scrapboxは「ノードは他人もいじれる」「エッジはただの無属性有向」
- このモデルは「ノードは他人はいじれない」「エッジは有属性有向」

### ステルスノード
自分にしか見えないノードをつくれる。これをステルスノードという。

主な用途

- ブックマーク
- フィルタリング
- メモ etc

以下例。

```
話題XについてA、B、C、Dさんが言及しているとする。

    (A's phrase)
        |
        |
        |
        V
    (Topic-X) <------(B's phrase)
      A  A
      |  |
      |  +-----------(D's phrase)
      |
    (C's phrase)


Aさんは、BCDさんの意見を咀嚼したいとする。
```

### カスケード
ノードAにノード1,2,3,4,5が繋がっているとする。

このとき、Aをカスケードすると、Aの見た目はA(+5)みたいになる。つまり直接繋がったノードを重ねることで見た目を「nノードが散らばっている」から「1ノードに重なっている」にする。重なった見た目があるので（1,2,3,4,5の存在を）見失うリスクは低い。

ネットワーク視覚化のテクニックかも。たぶん二次元表示になるだろうから。

## about コミュニケーションスタッキング、use 雑談タイム

### what is コミュニケーションスタッキング
コミュニケーションの機会を積むことで親密・友好・単純接触効果・お互いの把握などを高めることを **コミュニケーションスタッキング(Communication Stacking)** と呼ぶことにする。長いのでコミュスタと略す。

### コミュスタの問題点
コミュスタは仕事において重要とされているが、あらゆる会議において「ついでに」行われるのが現状である。そしてコミュスタ自体は効率や生産性とは無縁のものである（極端な話、家族との交流にそれらを求めようとはすまい）。

よって、会議そのものがその（無縁という）性質に引きずられて、つまりは非効率的で非生産的になってしまう。定例会議、全員参加、半強制的なビデオオン――思い当たる節はあろう。

問題に拍車をかけるのは、コミュスタの必然性を疑っていない勢力の存在である。筆者の観測範囲内（主に何十年と続く万単位従業員の大企業）で申し訳ないが、40代以上の者はこの傾向が強く、withコロナであっても対面コミュニケーションやビデオ付きの会議を好む傾向にある。「一週間対面会議を全く行わないフルリモート」のような働き方もまず受け付けない。無論、若手の層にもこの勢力は存在し、「毎日定例会議を開催して顔を合わせるチーム」も珍しくはない。

### コミュスタを分離する
会議をより効率的・より生産的にするには、コミュスタの側面を切り離す必要がある。つまり「会議を行うための時間」と「コミュスタを行うための時間」とに **分離** し、前者には後者を混ぜない。

そもそもコミュスタそのものは効率・生産性とは無縁の営みである。このような営みを **CWEP(Communication Without Efficiency and Productivity)** と呼ぶことにするが、要するに会議に CWEP を混ぜないようにするのである。

無論、人間である以上、CWEP そのものは必要であるが、それは CWEP 専用の時間を別途確保することで担える(※1)

※1 beforeコロナであれば、オフィスに全員が集まっておりアトランダムに雑談が発生していたため CWEP は行われていた。また飲み会や喫煙室での井戸端会議などもあった。withコロナではそれができないため、別の方法を考える必要がある。それが「CWEP専用の時間を設ける」である(※2)。

※2 beforeコロナだからといって全員が CWEP の恩恵を受けていたかというと、そうではない。人付き合いが苦手な者や下手な者だっている。かくいう筆者もその口で、よくチームでは「筆者以外の全員がランチに行っている」「会議に参加している」という状況が起こっていた。beforeコロナ時代の CWEP は、よくも悪くもある種の要領が求められる。withコロナになってからは、その要領がないものでも仕事がしやすくなった……と小話をしてしまったが、要するに before コロナ時代の CWEP が最適だったわけでもない。

### 雑談タイム(概念レベル)
CWEP 専用の時間として具体的に何を行えばよいか。アイデアはいくつも考えられる（というよりまだ誰も本格的に追求してはいないだろうからこれから考えていくことになる）が、筆者のアイデアを一つだけ挙げてみよう。

**雑談タイム** とは、雑談するためだけの時間である。

- 毎日10～30分、確保する
- 業務時間中に確保する
- 雑談タイム中は、仕事の話はしない
    - 自然な流れで行うのは良い
    - 命令や圧力の形で半ば強引に仕事の時間に変えるのはナシである
- 定時間前に行う
    - 雑談タイム後、用がなければ帰宅できる

ポイントは「仕事の一環として」行うこと、しかし「仕事の話はしない」ことにあろう。一見すると矛盾しており、真面目な者は頭ごなしに否定しがちである。

既に見たように、コミュスタは人間である以上必要でありながら、「ついでに」行われるがゆえに効率と生産性が削がれていた。両方に切り込むには、コミュスタを分離し、コミュスタ用の時間を設けるのが理にかなっている。

### 雑談タイムのやり方
と、コンセプトについては述べたが、実はまだ具体的なやり方までは整備できていない。

- 何人で行うか
- 誰と誰が喋るのか
- メンバーは毎回固定するのか、それともローテーションか、シャッフルか
- いつ、誰が、何について喋るのか
- 従来の CWEP の問題はどうするのか
    - シャイが集まるとコミュニケーションが進みづらいこと
    - 声の大きい者や権力を持つ者による「一方的な講釈の場と化する現象」
    - 喋るのが苦手な者
- etc

筆者が考えているアプローチは２つほどある。

まずは **1on1** である。

1on1 の起源は定かではないが、元々は「1on1 coaching session」として整備されたビジネスフレームワークであり、2017年頃に[ヤフーの取り組み](https://www.amazon.co.jp/dp/4478069786)にフォーカスが当たったことでブームが加速した印象がある。1on1 というと上司と部下間で定期的に行うミーティング、というニュアンスが強い。

しかし最近はそうでもなく、クロス1on1、いきなり1on1、斜め1on1、シャッフル1on1など、多様な広がりを見せている。共通しているのは、社内の色んな人との雑談機会を増やそうという意思であろう。関連するサービスは既にいくつも展開されている。

1on1は「コミュスタに特化した時間」と言えるだろう。今のところは「せっかくの貴重な時間なのだからビジネスやキャリアに役立てる」という意識が強いようだが、それ以外の目的――ここで筆者が主張する「単にコミュスタを行うための時間」も含めて――も模索されるだろう。

多様な 1on1 文化の広がりとともに、雑談タイムのやり方が開拓されていけば良いなと筆者は考えている。また、積極的にヒントとして活用して、筆者自らやり方を開拓したいとも思っている。

それからもう一つが **ゲーム** である。

読者はスプラトゥーンをご存知だろうか。ポップなキャラクターがペンキ銃を撃ち合う、あのゲームである。エンジニア界隈ではずいぶんと人気なようで、Twitter で遊んでいる光景をよく見る。筆者は全くの無知であるが、あのようなゲームを通じてでもコミュスタは行えるのではないかと思っている。

そうでなくとも、たとえばボードゲームなどはゲームを楽しむためではなく他者と触れ合いたいがゆえに行われていたりする(※1)。既に（ゲームというよりは「講師」によるワークショップだが）[バヅクリ - テレワーク時代のチームビルディング](https://buzzkuri.com/) のようなサービスも登場している。

要するに、ゲームはコミュスタを行うための手段として重宝するのではないかと筆者は考えている。なにせ皆のやることが（ゲームという形で）決まっているため、当人同士で話題を考えたり空気を読んだりしなくても良い。システマチックに過ごせるのである。

ひょっとすると、未来のビジネスパーソンは1日30分、チームメンバーとゲームをして過ごしているかもしれない――そう考えるのは妄想だろうか。

## about 投票ベースの議論、use Wikisurvey
投票ベースの議論とは

- 議題に対して賛成反対を集めたい

Wikisurveyとは

- [The Computational Democracy Project - The Computational Democracy Project](https://compdemocracy.org/Wikisurvey/)
- インタラクティブな投票方式の一つ
- 特徴
    - 参加者は全ての質問に回答しなくてもいい
    - 参加者は（他の参加者全員が回答する）質問を新たに追加することができる
    - 回答数や質問数は結果に左右される
- 参加者が自分の好きな程度で参加できること、誰でも観点を追加できること、参加すればするほど高密度かつ具体的になっていくこと

これを実装したシステムとして Pol.is がある

- 議題がある
    - 議題には n の論点がある
    - 論点にはyes, no, わからないを答える
- 参加者は議題に参加し、nの論点すべてに回答する
    - 新たな論点を追加することもできる
    - 「新たに論点が追加されたらメールで知らせる」的な機能もあるので、後から追加されても答えられる
- 議題内の回答傾向や分布はビジュアライズされる
