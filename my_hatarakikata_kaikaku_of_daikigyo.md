# この文書について
大手 IT 企業に勤務する筆者の持論について雑多にまとめている。

# 社内 Q&A システム(inqanda)
社内 Q&A システム(inqanda)とはイントラネット内に構築した Q&A システム(qanda)のこと。

現状、大企業は「社内情報や社内固有事情の検索・入手が非効率」という問題を抱えている。これを可能な限り軽減できる、最も強力で、しかし現実的な手段が inqanda である。

## 要旨
- 情報検索は「ググってすぐ見つかる」レベルで素早く済むのが理想である
- しかし、社内でこれを実現するのは非現実的である(Google の技術を再現するのは難しい)
- 代案として、社内で Q&A サイトを構築する
  - 全社全員が誰でも閲覧・質問・回答・評価できる
  - 評価の高い質問 = 価値のある質問なので、検索時は優先的に表示する
  - Q&A サイトを活発化させるための普及、啓蒙、ゲーミフィケーション等は必須である

## 背景
社内情報を探したり疑問を解消したりするのに多大な手間暇がかかっている。

- 人に訊く(忙しくて捕まらなかったり「ちょっと待っててね」と Pending が起きたりする)
- 分厚く読みづらいマニュアルを読む
- 社内ポータルや共有フォルダを辿る
- スタッフ部門に質問する(手段も古典的なメール or 電話)

一方で、インターネットでは Google 検索をしたり qanda に質問したりすることで素早く解決できる。社内でもこのレベルに近づけたい。

### 社内情報の例
- 出張申請システムのこの欄には何を入力すればいい？
- フリーソフトの利用申請はどうすればいい？
- 病欠休暇制度について知りたいんだけど、どこ見ればいいんだっけ？
- 新しいディスプレイを調達したいんですが、どうすればいい？
- 部署異動したんですが社内システム A が反映されません、どうすればいいんですかこれ？

## システム概要

### ターゲット
全国に拠点を持ち、従業員が数千を超えるような大企業または企業グループ

### 欠かせない要件
- 全社員全員がいつでもアクセスできること
- 全社員全員が質問、回答、(質問や回答に対する)評価を行えること
- 全社員全員がプロフィールページを持ち、自由に編集できること
- ゲーミフィケーションをサポートしていること
- 質問・回答の検索には評価数フィルターを設けること

### コンセプト
Stack Overflow のような qanda。

- 良質な質問回答が集まるようにガイドラインとメンテナを設け日々運用する
- 評価機能により質問回答に価値を付け、検索時に価値の高いものを表示させることで検索精度を担保する
- ゲーミフィケーションにより回答者評価者の意欲を高め、有益な情報が集まるように保つ
- 数の力を結集してナレッジデータベース(KDB)を実現する
  - Stack Overflow はワールドワイド
  - ターゲット企業は数千以上の人数だが、これは結集による KDB を実現できる最低人数
  - これより少ないと利用者数の絶対数が少なすぎて発展しない

理想は Google 検索のように「ググって一発で見つかる」ことだが、技術的に非現実的であるため qanda で妥協する。

## inqanda 実現に必要なもの・こと
- inqanda に耐えうるインフラ
- inqanda をオンプレで実現するための qanda ソフトウェア
- qanda に関するリテラシー教育

### インフラ
全社員全員がいつでもアクセスできるのに耐えうる程度は欲しい。

### qanda ソフトウェア
これといった決め手に欠ける印象。

現状の実用的な選択肢は以下くらいだと思われるが、 ~~ターゲット企業レベルの規模での利用例が無い~~ あった。[Yahoo! Japan では 7000 人の全社員が利用で1日1.5K投稿](https://seleck.cc/486)

- [Questions for Confluence - アトラシアン](https://ja.atlassian.com/software/confluence/questions)
  - アトラシアンのナレッジ管理ツール Confluence のアドオン
  - 参考: [Confluence Questions(ナレッジ管理、FAQツール) - リックソフト](https://www.ricksoft.jp/atlassian/confluence-questions/)
- [Scoold - An Open Source Q&A Platform](https://scoold.com/)
  - Java 製の Stack Overflow クローン

無いなら新規開発するのもアリ。inqanda というジャンルは未だ開拓されていないが、大企業を救えるポテンシャルがあるので、開拓できれば良きメシの種になる。啓蒙やコンサル等もセットにすればさらに稼げる。

### 上記 Yahoo 記事まとめ:
> 私は全社のおよそ7,000名が使用している社内情報共有ツール「Confluence」の運用に、3名のチームで関わって

素晴らしい。3 名くらいでできそう、というのは私もそう思う。

> 1日におよそ15,000回もの投稿がなされるという。投稿される内容は会議の議事録からプロジェクトの進捗報告、社内広報など、近隣の美味しい料理屋マップまで、幅広い領域に及んでいるそうだ。

ちょっとイメージ違う。私が提案する inqanda よりも緩い。たぶん Yahoo は社内雑務が洗練されている(規模が少ない)から、その分雑談ネタや PJ 固有ネタも扱える余地があるのだと想像。Yahoo ほどリテラシーのない大企業ではこうもいかない。社内雑務の規模が大きいので、ストイックに実用的な質問を優先的に扱う必要がある(そうしないと雑談というノイズのせいで実用的な情報を素早く探せない)。

### qanda リテラシー教育
qanda の利用に成熟している者は少ない。明示的な教育は必須。必須教育にしても良いレベル。

教育内容:

- qanda の使い方
- 利用者が持つべきリテラシー全般
  - 質問回答評価のマナー
  - qanda のコンセプト
  - ガイドラインの内容
- ……

## ゲーミフィケーション
inqanda を実現する上での肝。

ナレッジを知る者に回答させるためには相応の報酬が必要。内発的な報酬と外発的な報酬の二つがある。

- 内発的な報酬
  - 承認や貢献のこと
  - バッヂ、レベル、ランク、ベストアンサー数/率
  - ユーザーランキング(部署内、事業部内、拠点内、全国)
- 外発的な報酬
  - 金銭報酬や待遇報酬のこと
  - 上位 n 位に金銭報酬
  - qander(qandaにこもって回答やメンテを生業とする人)という肩書や仕事を用意して専任させる
  - 上位 qander は部長級の給与待遇にする
  - qander は毎日定時の範囲内で動ける働きやすい職場 ← これをアピールしたら食いつく

## 質問として扱うべき内容
- 扱う
  - 仕事上、実用的な内容
  - 仕事のやり方、立ち振舞い、勉強方法など仕事に関する相談
- 扱わない
  - 時事ネタ、教養ネタ全般(Quoraでしばしば見られる)
  - ユーモアを目的としたネタ全般(Yahoo! 知恵袋でしばしば見られる)
  - 仕事と関係のない、私生活に関するネタ全般(発言小町でしばしば見られる)

ニュアンスとしては Stack Overflow が最も近い。Stack Overflow はプログラミングに関する話題に限定している。これの社内・仕事話題限定版。

Quora や Yahoo! 知恵袋のような娯楽ネタは扱わない。ここを扱い始めると質問回答の大半がこれらで固まってしまい、実用的な利便性や価値が薄まってしまう。

## 評価
様々な方法が考えられるが、まだどの方法が良いかは思いついてない。

ブレスト:

- Stack Overflow では vote up/down
- Yahoo! 知恵袋ではベストアンサー
- (qandaではないが)Amazon では ☆ 五段階
- 「n時間助かりました」の n を入力 or 選択させる

# フリーソフト利活用を推進するためのフリーソフト管理案

## 現状の問題点
- 申請に要する負担が大きい(重たい Excel 操作、多すぎる申請項目、部門長一人に集中する承認)
- 検索に要する負担が大きい(重たい Excel、Word、PDF を読む必要がある)
- フリーソフトの効用が認知されていない(特に「個人的な作業効率/生産性/PC利用時のストレスを減らせること」)
- フリーソフトに対する誤解がはこびっている
  - 例1: 別に使わなくても仕事できるでしょ
  - 例2: フリーソフトは危ないよね
  - 例3: フリーソフトは無料だし品質も悪いし会社で使うのはダメだよね
- 管理が部門単位であるため部門毎に管理作業が重複している

## 理想
- 部門毎の管理を撤廃し、全社一律で管理している
- 「フリーソフト管理システム」が存在する
- 「フリーソフト申請前判定システム」が存在する
- 会社はこれら知見やノウハウを生かして他のビジネスやアピールに応用している
- 役職「フリーソフトエバンジェリスト」が存在する

### 部門管理 → 全社管理
大企業におけるフリーソフト管理は部門毎の独自管理になっていることが多い。

- 理由: 大企業と部署の関係は、国と村の関係のようなものだから
  - 村それぞれが閉鎖的になりがちなように、部署もまた閉鎖的になりがち

しかし部門毎の管理には以下問題がある。

- 管理作業が部門間で重複しており無駄
- フリーソフトの知見が部門を超えて共有されない
- 承認者(≒部門長)が保守的であるがゆえに、フリーソフト利用緩和に向けて動こうとしない
  - 承認者は現状の厳しいフリーソフトルールでも特に困っていないことが多い
  - もし利用緩和すれば、この先問題が起きるかもしれない
  - 問題が起きたら自分の評価にも響く
  - → リスクのあることをわざわざやる必要はない

無駄を省き、共有を実現し、承認者への責任集中をするためにも、部門毎の管理から全社一律の管理に押し上げる必要がある。

### フリーソフト管理システム
全社全員誰でもいつでもアクセスできる。

申請:

- 必要なのは「ソフト名」「入手先URL」のみ
- 申請内容については全社全員誰でもコメントできる(数の知による是非判定)
- 申請内容はデフォで自動承認。問題がある場合のみ(承認部門が)否認操作を行う
- 一度承認されたものは永続利用可(半年ごとに再申請だなんて不毛なことはしない)

閲覧:

- フリーソフト一件ごとに専用ページがある
- コメント、評価、承認状況などがある

検索:

- 評価の高い順、コメントの多い順、承認日時順などでソート可

### フリーソフト申請前判定システム
- ソフト名を入れると OK/NG/要申請/申請済 を出力してくれるツール
- ブラウザから利用できる

### フリーソフトのビジネス化
本案を推進してフリーソフトに関する知見や実績を蓄積することができれば、さらなるビジネスのネタとして活用することもできる。

- 例1: お客さんに向けた情報発信やコンサル
- 例2: 当社はフリーソフト利活用で～～(インパクトのある定量的指標)の効果を実現しました！

### フリーソフトエバンジェリスト
フリーソフトエバンジェリストとは、社内フリーソフト利用を推進するための社内第一人者である。

活動例:

- フリーソフト管理システム等での積極的な活動(Stackoverflow のような Q&A サイトのメンテナみたいな感じ)
- 全社全員誰でも閲覧＆コメント可能なブログを運営し、定期的に情報発信
- 部長レベルの給与待遇

本案の推進にあたっては、詳しい者(筆者もその一人である)に権限と裁量を与えて任せてしまうのが良い(★理由書いて)が、推進には多大な労力と気力を要するため、相応の立場、待遇、報酬を用意する必要がある。

## フリーソフトの効用
- 業務上必要な作業を n 時間/日 短縮できる
  - 例1: ファイル比較は目視よりも WinMerge
  - 例2: ファイル検索は手作業よりも Grep(たとえば秀丸エディタ)
  - 例3: スクリーンショットは PrintScreen + Mspaint よりも WinShot
- 個人の PC 操作を効率化して操作を素早く行えるようにできる＆ストレスを軽減できる
  - 例1: ブラウザは動作が遅く拡張性も乏しい IE よりも Chrome や Firefox
  - 例2: 多数のファイルを素早く呼び出すためのプログラムランチャ
  - 例3: 使いづらいキーアサインを変更するために AutoHotkey
- PC 利用に関するリテラシーが鍛えられる
  - 例1: 改善のために自ら設定やツールを変えたり探したりする姿勢
  - 例2: ファイル、フォルダ、レジストリ、設定ファイルといった基本素養の習熟

★効用は「直接定量的な成果が出る」よりも「定性的に個人を改善＆パワーアップさせる」のがメインなのでアピールしづらい……

## フリーソフトに関する誤解
★以下あたりを詳しく書きたい
- あるのとないのとでは大違い（効用の話と被る）
- フリーソフトは完全に安全とは言えない(そもそも何事にも 100% の安全はない)が、やり方を少し気をつければ安全に利用できる
  - 海賊サイトは使わないとか、その辺我々エンジニアが当たり前にやってることを明文化する感じだろうか
- フリーソフトの品質はピンキリだが、知られているものは社内で自製したものよりもはるかに優れていることが多い

## 現状のフリーソフト管理ルール
詳細は企業次第だが、概ね以下のようなルールに収束している。

- 全社的ルールと部門裁量
  - 全社的ルールではホワイトリストとブラックリストを定める
    - ホワイトリストにあるもの → 各自勝手に使っても良い
    - ブラックリストにあるもの → 問答無用で利用禁止
  - それ以外は部門単位での管理が主流
- 部門管理
  - 承認者(部門長など権限のある者)が承認すれば利用可能になる
  - 申請の過程は記録に残し、後で追えるようにする
  - 定期的に棚卸し、または全リセットからの再申請にする
  
- それ以外 → 部門のプログラム管理責任者(部門長もしくは部門長から委譲された者)の承認後、使っても良い
  - ただし申請承認内容は記録に残す

# 画一化の撤廃と部分的優遇
大企業では全社員を一律して「特定の社員像」で捉え(画一化)、この画一化された社員像をベースに各種制度を整える。しかしながら、この社員像はリテラシーの低い側に合わせられるために、リテラシーの高い側は不便を強いられることになる。

仕事の成果や生産性の大部分が「リテラシーの高い側」によってもたらされることを前提とすると、彼らに不便を強いることが大きなダメージとなってしまうことは想像に難くない。このダメージを防ぐのが本案であり、本案では画一化の撤廃を提案する。

## 部分的優遇

### Step1. 分類
画一化を撤廃した上で、以下のように分類する。

- クリエイター
- 非クリエイター

例をいくつか挙げる。

- 例1: IT 企業
  - エンジニア
  - 非エンジニア
- 例2: 営業メインの企業
  - セールスマン
  - 非セールスマン
- 例3: デザイン会社
  - デザイナー
  - 非デザイナー

### Step2. クリエイターの選定
全社員をクリエイターと非クリエイターに分類する。

★どうやって分けるかが難しい。企業はわかってないから企業主導にすれば不幸なアンマッチになる。志願制にしたら関係無い人まで志願してしまう。審査が良い？（たとえば github、qiita、twitter アカウントを持って活動している等）……でも企業にバラすのはしんどいよな。

### Step3. クリエイターには別制度を
非クリエイターには従来どおり画一的な制度を与え、クリエイターには全く別の制度を与える。つまり制度は二つ存在する。

ただし現実的には「画一的な制度」をコピーして、部分的に修正した制度を使うことになるだろう。

## エンジニアの優遇
私はエンジニアであり、ソフトウェア企業に属しているため、以下「エンジニア」「非エンジニア」という分け方を前提として議論する。

### 何を優遇するか
- 利用するハードの緩和・解禁
  - 例1: 標準的な Win10 + 4GB + HHD → Win10 + 16GB + SSD
  - 例2: 標準的なキーボードとマウス → 好きなキーボードとマウス
  - 例3: 標準的なディプレイ 1 枚 → 好きなディスプレイ 2 枚
  - 例4: 量産された事務椅子 → デザイナーズチェア
- 利用するソフトの緩和・解禁
  - フリーソフト
  - OSS
  - クラウドツール
- 勤務体系の緩和・解禁
  - 9-18 勤務、のように固定的に縛らない
- コミュニケーション手段の配慮・緩和・解禁
  - 安易に口頭に巻き込まない(口頭は最もコストのかかる手段)
  - キューイングによる処理を基本にする
    - 発信者は発信しておく
    - 受信者は自分のタイミングで確認＆返信する
  - ★エンジニアの感性、観点、その辺まとめたいか？
